—— FUNDAMENTALS

Containers Are Lightweight, Standalone Packages That Include Everything Needed to Run

Code, runtime, system tools, libraries, and settings — all bundled into a single, portable unit that runs the same way everywhere.
  • Process isolation — containers share the host OS kernel but run in their own isolated space
  • Consistent environments — identical behavior from development through production
  • Fast startup — seconds instead of minutes, with minimal resource overhead
—— COMPARISON

Virtual Machines vs. Containers

Virtual Machines

Include a full guest OS, hypervisor layer, and dedicated resources. Strong isolation but heavier resource footprint and slower startup times measured in minutes.

Containers

Share the host OS kernel and isolate at the process level. Lightweight, start in seconds, and enable much higher density on the same hardware.

—— ARCHITECTURE

Three Core Building Blocks

Images

Read-only templates with application code, runtime, and dependencies layered together.

Containers

Running instances of images with a writable layer for application state and data.

Registries

Storage and distribution systems for container images like Docker Hub or Quay.io.

—— WORKFLOW

The Container Lifecycle: Build → Ship → Run

Three steps take you from source code to a running application — anywhere.

1. Build

Write a Containerfile that defines your app's environment. Build it into an immutable image with podman build.

2. Ship

Push the image to a registry (Quay.io, Docker Hub, or private). Any environment can pull it — dev, staging, or production.

3. Run

Start a container from the image with podman run. The same image produces identical behavior every time, everywhere.

—— HOW IMAGES WORK

Every Instruction Creates a Cached Layer

Container images are built from stacked, read-only layers. Each line in your Containerfile adds a layer. Unchanged layers are cached, making rebuilds fast.
# Layer 1 — base image (cached after first pull) FROM docker.io/library/nginx:alpine # Layer 2 — remove default page (cached) RUN rm /usr/share/nginx/html/index.html # Layer 3 — your code (rebuilds when file changes) COPY index.html /usr/share/nginx/html/ # Metadata only — no new layer EXPOSE 80 CMD ["nginx", "-g", "daemon off;"]
  • Fast rebuilds — only changed layers are rebuilt; everything above the change is cached
  • Smaller images — fewer layers and cleanup in the same RUN step keeps images lean

—— ORCHESTRATION

Running Containers in Production Requires Automated Management

When you move beyond a single container, you need orchestration for deployment, scaling, networking, and availability across clusters of machines.
  • Automated scheduling and placement of containers across nodes
  • Self-healing with automatic restarts and replacements
  • Service discovery and load balancing built in
  • Rolling updates and automated rollbacks

—— BEST PRACTICES

Container Best Practices

Security

Use minimal base images, scan for vulnerabilities, and run as non-root.

Logging

Send logs to stdout/stderr and aggregate with centralized logging.

Resources

Set CPU and memory limits to prevent runaway containers.

Image Size

Keep images small with multi-stage builds. Remove unnecessary files.

Health Checks

Implement liveness and readiness probes for automated recovery.

Immutability

Never patch in place — rebuild and redeploy for any change.

—— TOOLING

Podman: A Drop-In Replacement for Docker

If you know Docker, you already know Podman. Same CLI, same Containerfile format — but with key architectural differences that matter in production.

Docker

Requires a privileged root daemon (dockerd) running in the background. All container operations go through this single process. If the daemon stops, all containers stop.

Podman

Daemonless — each container runs as a direct child process. No root service required. Rootless by default. Same CLI commands: podman build, podman run, podman push.

  • Same CLI — alias docker=podman and existing scripts just work
  • Red Hat default — ships with RHEL and is the standard tool in the OpenShift ecosystem
—— SECURITY

Rootless Containers Eliminate Privilege Escalation

Traditional containers run as root on the host — if an attacker escapes the container, they own the machine. Rootless containers solve this by running entirely under a normal user account.

Root Containers

Container processes run as root on the host. A container escape grants full host access. Requires a privileged daemon (e.g., Docker's dockerd).

Rootless Containers

Container processes map to an unprivileged user on the host. Even a container escape yields zero host-level privileges. No privileged daemon required.

  • OpenShift ROSA default — ROSA enforces rootless containers out of the box via Security Context Constraints
  • Podman is rootless-native — no daemon, no sudo — rootless is the default mode, not an add-on
  • Defense in depth — user namespaces isolate container UIDs from host UIDs at the kernel level
